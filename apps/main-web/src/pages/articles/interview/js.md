# JavaScript 面经

## 函数缓存

### 1. 基本原理

函数缓存是一种空间换时间的优化手段。其核心逻辑是将函数的输入参数作为键（Key），将执行结果作为值（Value）存储在闭包内的 Map 或 Object 中。当下次以相同参数调用该函数时，不再执行函数体内的复杂逻辑，而是直接从内存中返回已存储的结果。

### 2. 核心代码实现

在实际开发中，通常会封装一个高阶函数来实现通用化的缓存机制：

```javascript
function memoize(fn) {
  const cache = new Map(); // 使用 Map 存储缓存，支持更多类型的 Key
  return function (...args) {
    const key = JSON.stringify(args); // 将参数序列化作为唯一标识
    if (cache.has(key)) {
      return cache.get(key); // 命中缓存，直接返回
    }
    const result = fn.apply(this, args); // 执行原函数
    cache.set(key, result); // 存入缓存
    return result;
  };
}
```

### 3. Vue 场景下的典型应用

- **计算属性（Computed）**：这是 Vue 内部最典型的函数缓存实践。Vue 会根据响应式依赖自动缓存计算结果，只有当依赖项发生变化时，才会重新求值。
- **昂贵的递归计算**：在处理像斐波那契数列、大规模树形数据转换或复杂数学公式计算时，通过缓存避免重复渲染过程中的冗余计算。
- **API 请求过滤**：对于短时间内参数完全相同的查询类接口（如配置信息获取），可以在前端层做函数缓存，减少不必要的网络开销。

### 4. 关键点与边界检查

- **纯函数前提**：函数缓存只适用于“纯函数”（即相同输入必有相同输出，且无副作用）。如果函数依赖外部环境变量或具有随机性，则不能使用缓存。
- **内存管理**：缓存会持续占用内存。如果参数组合极其多样，会导致缓存对象无限膨胀。在大型项目中，建议引入 LRU（最近最少使用）算法来自动清理旧缓存。
- **参数序列化成本**：对于参数对象极其庞大的函数，序列化（JSON.stringify）本身的性能损耗可能超过函数执行本身。此时应考虑使用哈希算法或自定义 Key 生成逻辑。

## 高阶函数

### 1. 柯里化 (Currying)

**本质：** 将一个接收多个参数的函数，转化为一系列接收单个参数的函数。其核心利用了闭包特性。

**技术实现：**

```javascript
// 基础：add(1, 2, 3) -> curryAdd(1)(2)(3)
const curry = (fn) => {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn.apply(this, args);
    }
    return (...nextArgs) => curried.apply(this, [...args, ...nextArgs]);
  };
};
```

**工程价值：**

- **参数复用：** 比如在 Vue 项目中封装通用校验逻辑或权限判断，可以固定部分公共参数（如权限等级），生成特定用途的定制函数。

- **延迟执行：** 逐步收集参数，直到满足条件才触发逻辑。

### 2. 函数组合 (Composition)

**本质：** 将多个单一职责的小函数通过从右向左（Right-to-Left）的方式嵌套执行，前一个函数的输出作为后一个函数的输入。

**技术实现：**

```javascript
// 逻辑：f(g(h(x)))
const compose =
  (...fns) =>
  (x) =>
    fns.reduceRight((v, f) => f(v), x);

// 示例
const toUpper = (s) => s.toUpperCase();
const addExclamation = (s) => s + '!';
const greet = compose(addExclamation, toUpper); // 先转大写，再加感叹号
```

**工程价值：**

- **声明式编程：** 隐藏执行细节，直接描述“做什么”，提高代码的可读性。

- **解耦逻辑：** 鼓励编写单一职责的纯函数，便于单元测试和代码复用。

### 3. 函数管道 (Piping)

**本质：** 逻辑与组合一致，但执行顺序改为从左向右（Left-to-Right），更符合人类的线性思维习惯（类似于 Linux 的管道符 `|`）。

**技术实现：**

```javascript
// 逻辑：h(g(f(x)))
const pipe =
  (...fns) =>
  (x) =>
    fns.reduce((v, f) => f(v), x);
```

**工程价值：**

- **数据流清晰：** 在处理复杂业务逻辑（如表单数据清洗、格式化输出）时，管道化让数据转换的过程一目了然。

### 4. 核心对比与实践总结

| 概念       | 执行顺序 | 核心目的          | 常见场景                        |
| :--------- | :------- | :---------------- | :------------------------------ |
| **柯里化** | 参数降维 | 固定环境/预设参数 | 绑定 API 配置、Vue 指令参数预设 |
| **组合**   | 从右向左 | 嵌套逻辑扁平化    | 函数式编程库（Lodash/fp）       |
| **管道**   | 从左向右 | 线性数据加工流程  | 复杂数据预处理、多步转换逻辑    |

**在 Vue 场景下的习惯：**
在编写 Composable (组合式函数) 时，经常会将逻辑拆分为细小的 Helper 函数。如果逻辑是线性的（如：获取列表 -> 过滤数据 -> 排序 -> 映射字段），使用 **Pipe** 结构能让维护者迅速理解数据处理链条，而不是深陷嵌套的括号中。

## 函数式编程

### 1. 核心思想

函数式编程是一种将计算视为数学函数计算的范式，强调“做什么”而非“怎么做”（声明式优于命令式）。其核心是将逻辑拆解为一系列可复用、可组合的纯函数，通过组合这些函数来解决复杂的业务逻辑，从而提高代码的可读性和可维护性。

### 2. 三大基本准则

- **纯函数 (Pure Function)**：对于相同的输入，永远得到相同的输出，且没有任何副作用（不修改外部变量、不发起异步请求等）。这使得逻辑极易测试和预测。
- **不可变性 (Immutability)**：数据一旦创建就不直接修改。若需更新，则返回一个包含新值的新对象。这在状态管理中能有效避免“数据被谁改了”的追踪难题。
- **函数是一等公民 (First-class Function)**：函数可以像变量一样被赋值、作为参数传递或作为返回值输出。

### 3. 常用技术手段

- **高阶函数 (Higher-order Function)**：接收函数作为参数或返回函数的函数。例如 JavaScript 原生的 map、filter、reduce。
- **函数组合 (Compose)**：将多个单一职责的小函数通过“管道”连接，形成一个功能复杂的函数。
- **柯里化 (Currying)**：将一个多参数函数转化为一系列单参数函数的过程。

```javascript
// 柯里化示例：用于权限判断或环境配置
const checkRole = (role) => (user) => user.role === role;
const isAdmin = checkRole('admin');

const user = { name: '张三', role: 'admin' };
console.log(isAdmin(user)); // true
```

### 4. 在 Vue 框架中的实践体现

- **Composition API (Vue 3)**：本质上是函数式编程的回归。通过把逻辑封装在独立的 `useXxx` 函数中，通过解构和组合来构建组件逻辑，解决了 Mixin 的命名冲突和来源不明问题。
- **计算属性 (Computed)**：典型的纯函数应用。它根据依赖项计算结果，不修改原始数据，只返回新值。
- **数据流管理**：在 Vuex 或 Pinia 中，强调 State 的不可变性，必须通过 Action 或 Mutation 这种确定性的方式来变更状态，这正是函数式编程中“单向数据流”和“可预测性”的体现。

### 5. 优缺点评估

- **优势**：逻辑高度解耦，易于单元测试；由于没有共享状态，极大减少了 Bug 的产生；天然适合并发环境（虽然 JS 是单线程，但在处理复杂逻辑流时更清晰）。
- **挑战**：对习惯了过程式思维的开发者有一定学习成本；过度使用闭包和频繁创建新对象可能带来微小的内存压力。

## 本地存储方式比较

### 1. 核心存储方式对比表

| 特性         | Cookie                               | LocalStorage           | SessionStorage                   | IndexedDB                  |
| :----------- | :----------------------------------- | :--------------------- | :------------------------------- | :------------------------- |
| 存储大小     | 约 4KB                               | 约 5MB                 | 约 5MB                           | 磁盘剩余空间的 80% 左右    |
| 生命周期     | 可设置过期时间，否则随浏览器关闭失效 | 永久存储，除非手动删除 | 页面会话结束（关闭标签页）即失效 | 永久存储，除非手动删除     |
| 与服务端通信 | 每次请求自动携带在 HTTP 头中         | 不参与                 | 不参与                           | 不参与                     |
| 数据格式     | 字符串（需 key=value 格式）          | 字符串                 | 字符串                           | 结构化对象（支持索引搜索） |
| 读写性能     | 较快                                 | 较快（同步）           | 较快（同步）                     | 较慢（异步操作）           |

### 2. 各存储方式详解

- **Cookie**
  最初设计用于标识用户身份。由于每次请求都会携带，存储过多会严重影响性能。通常配合 HttpOnly 属性使用以防止 XSS 攻击，确保安全性。

- **LocalStorage**
  属于持久化存储。在实际开发中，常用于存储不易变动的配置项，如用户的主题偏好（黑夜模式）、多语言设置或长期缓存的静态数据。

- **SessionStorage**
  其生命周期严格限制在标签页内。在 Vue 项目中，常用于存储表单录入过程中的中间状态，或单次登录后的敏感 Token（不希望持久化在磁盘），确保关闭页面即清理。

- **IndexedDB**
  一种运行在浏览器端的非关系型数据库。适用于离线应用（PWA）或需要存储大量结构化数据（如本地离线地图数据、草稿箱缓存）的场景。

### 3. 代码示例与最佳实践

在 Vue 开发中，针对 LocalStorage 的读写通常会封装工具类，以处理 JSON 序列化带来的异常：

```javascript
const storage = {
  set(key, value) {
    // 自动处理对象序列化
    window.localStorage.setItem(key, JSON.stringify(value));
  },
  get(key) {
    const val = window.localStorage.getItem(key);
    try {
      return JSON.parse(val);
    } catch (e) {
      return val;
    }
  },
  remove(key) {
    window.localStorage.removeItem(key);
  },
};
```

### 4. 选型建议

- **身份凭证**：优先使用 Cookie，并开启 HttpOnly 和 SameSite 属性规避风险。
- **跨页面持久数据**：选择 LocalStorage。
- **单次会话临时数据**：选择 SessionStorage。
- **大数据量或二进制文件**：选择 IndexedDB。

## JS 内存管理

### 1. 内存生命周期

无论是哪种编程语言，内存生命周期基本都包含三个阶段：

- 分配内存：声明变量、函数、对象时，系统会自动分配内存。
- 使用内存：对内存进行读写操作（即读写变量）。
- 释放内存：当内存不再需要时，由垃圾回收机制（GC）释放。

### 2. 垃圾回收机制（GC）

现代浏览器主要采用“可达性分析算法”来判断对象是否存活：

- 核心原理：从根对象（Root，如 window 或 global）出发，遍历所有引用链路。如果一个对象无法通过任何链路被访问到，则被视为“不可达”，会被标记并回收。
- 引用计数（已淘汰）：早期浏览器使用过，通过计算对象被引用的次数。由于无法解决“循环引用”问题（即 A 引用 B，B 引用 A，但两者都不再被外部需要），目前已不再是主流。

### 3. 常见的内存泄漏场景

在 Vue 或原生 JS 开发中，最容易导致内存无法被回收的情况包括：

- 意外的全局变量：在非严格模式下，未声明的变量会挂载到 window 上，导致无法释放。
- 未清理的定时器或回调：在组件销毁后，setInterval 或 setTimeout 仍在运行，且内部引用了组件的数据。
- 未脱离文档流的 DOM 引用：在 JS 中保存了 DOM 节点的引用，即使页面上该节点已被删除，只要 JS 引用还在，整个 DOM 树片段就不会被回收。
- 闭包监听：在全局对象或长生命周期对象中注册了闭包，导致闭包内部引用的变量一直留在内存中。
- 事件监听未移除：在 window 或 body 上绑定的 addEventListener，在组件卸载时若不手动 removeEventListener，会导致监听函数及闭包持续占用内存。

### 4. 代码层面的优化实践

- 及时手动解除引用：对于大型对象或数组，在使用完毕后将其赋值为 null。
- 利用 WeakMap 和 WeakSet：如果需要建立对象到数据的映射，优先使用弱引用。WeakMap 不会阻止垃圾回收器回收其键所指向的对象。
- 严格遵循生命周期：在 Vue 的 onUnmounted 钩子中执行所有清理逻辑。

```javascript
// 实践示例：防止内存泄漏
import { onMounted, onUnmounted, ref } from 'vue';

export default {
  setup() {
    const data = ref(null);
    const timer = ref(null);

    const handleResize = () => {
      /* 逻辑 */
    };

    onMounted(() => {
      window.addEventListener('resize', handleResize);
      timer.value = setInterval(() => {
        // 轮询逻辑
      }, 1000);
    });

    onUnmounted(() => {
      // 1. 移除全局事件监听
      window.removeEventListener('resize', handleResize);
      // 2. 清除定时器
      clearInterval(timer.value);
      // 3. 显式释放大数据对象引用
      data.value = null;
    });
  },
};
```

### 5. 内存监控工具

在面试中可以提到 Chrome DevTools 的使用：

- Memory 面板：通过 Heap Snapshot（堆快照）对比不同时刻的内存增量，定位未被释放的对象。
- Performance 面板：观察内存曲线，如果曲线呈阶梯状上升且不下降，则存在明显的内存泄漏。

## JS 浮点数精度丢失问题

### 1. 现象本质

在 JavaScript 中，0.1 + 0.2 并不等于 0.3，而是等于 0.30000000000000004。这种现象在涉及财务计算、进度条百分比或复杂数学运算时，会导致逻辑判断失效（如 a + b === 0.3 为 false）。

### 2. 产生原因

- 存储标准：JavaScript 遵循 IEEE 754 标准，使用 64 位双精度浮点数（Double Precision）存储数字。
- 进制转换：数字在计算机底层以二进制存储。十进制的 0.1 和 0.2 在转换为二进制时是无限循环小数。
- 舍入误差：由于 64 位存储空间有限（尾数位只有 52 位），计算机必须对无限循环的二进制进行截断或舍入。这种“在存储前就已经丢失精度”的二进制数，在运算后再转回十进制，就出现了可见的偏差。

### 3. 实际开发中的解决方案

方案一：整数化运算（最常用）
将小数放大为整数进行运算，得到结果后再缩小。这种方法适用于金额计算（如将“元”转为“分”）。

```javascript
// 计算 0.1 + 0.2
const add = (num1, num2) => {
  const m = Math.pow(10, 2); // 假设精度到小数点后两位
  return (num1 * m + num2 * m) / m;
};
```

方案二：使用 Number.EPSILON
这是 ES6 引入的最小常量，表示 1 与大于 1 的最小浮点数之间的差。用于误差范围内的相等判断。

```javascript
const equal = (a, b) => Math.abs(a - b) < Number.EPSILON;
console.log(equal(0.1 + 0.2, 0.3)); // true
```

方案三：toFixed() 格式化
如果只是为了展示结果，可以使用 toFixed(n) 进行四舍五入。

- 注意：toFixed() 返回的是字符串，且在某些边缘情况下存在“六舍五入”的不稳定表现，仅建议用于 UI 渲染层。

方案四：成熟的第三方类库
对于对精度要求极高的金融或科学计算场景，直接使用成熟的数学库。这些库在底层将数字转为大整数或字符串处理，彻底规避 IEEE 754 的限制。

- Big.js
- Decimal.js
- Math.js

方案五：BigInt（仅限整数）
如果涉及的是超出安全范围（Number.MAX_SAFE_INTEGER）的特大整数运算，应使用原生的 BigInt 类型，但在使用时不能混用普通数字，且不支持小数。

### 4. 总结建议

在前端开发中，面对金额必须保持高度敏感。原则是：展示用 toFixed，简单计算用整数倍率放大，复杂计算必须引入 Big.js 等工具库，严禁直接在业务逻辑中使用浮点数进行相等比较。

## 尾递归

### 1. 核心定义

尾递归是递归的一种特殊形式，指一个函数在执行过程中的最后一步操作是调用自身。由于调用出现在函数的末尾（Tail），且不再需要对递归返回的结果进行任何额外运算（如加法、乘法等），因此这种形式具备了被编译器或引擎优化的物理前提。

### 2. 结构对比（普通递归 vs 尾递归）

判断是否为尾递归的关键在于：递归调用是否是函数体中执行的最后一个动作。

- 普通递归：返回语句包含表达式运算，必须等待下层函数返回后才能完成计算。

```javascript
function factorial(n) {
  if (n === 1) return 1;
  return n * factorial(n - 1); // 存在乘法，不是尾递归
}
```

- 尾递归：通过增加参数（累加器）将中间结果传递下去，确保调用即结束。

```javascript
function factorial(n, total = 1) {
  if (n === 1) return total;
  return factorial(n - 1, n * total); // 最后一步仅调用自身，是尾递归
}
```

### 3. 尾调用优化（TCO）的原理

在常规递归中，每进入一层调用，系统都会在内存中创建一个“调用栈帧（Stack Frame）”来保存当前的变量和返回地址。如果递归层级过深，会直接导致栈溢出（Stack Overflow）。
而尾递归优化则是：由于最后一步不再需要当前栈帧的上下文，引擎可以直接覆盖当前的栈帧，或者复用同一个栈帧。这意味着无论递归多少次，栈的大小始终保持不变，内存开销从 O(n) 降低到了 O(1)。

### 4. JavaScript 中的支持现状

- 标准支持：ES6 规范明确规定了在严格模式（"use strict"）下，引擎应该支持尾调用优化。
- 引擎实现：目前只有 Safari 的 JavaScriptCore 引擎完整实现了这一优化。V8（Chrome/Node.js）虽然曾尝试实现，但由于调试困难和错误堆栈追踪（Stack Trace）丢失等复杂原因，默认并未开启。
- 开发建议：尽管引擎支持有限，但在处理深度递归逻辑（如无限级树结构遍历）时，编写尾递归风格的代码仍是最佳实践，这有助于逻辑清晰，并在必要时手动将其重构为迭代（While 循环）来彻底规避风险。

### 5. 总结：为什么要关注尾递归

在前端开发中，面对复杂数据结构处理或算法实现，尾递归提供了将“递归逻辑”转化为“常数级内存占用”的可能性。它是编写高性能、高鲁棒性代码的重要思维工具，也是评估开发者底层原理掌握程度的重要指标。
